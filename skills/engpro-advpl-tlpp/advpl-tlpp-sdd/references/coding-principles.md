# Coding Principles

Behavioral bias, not a checklist. Read before each implementation.

---

## Before Coding

- State assumptions explicitly. If uncertain, ask.
- Multiple interpretations exist? Present all — don't silently pick one.
- Is there a simpler approach? Say so. Disagree when necessary.
- Something unclear? Stop. Name what's confusing. Ask.
- Does the user's approach seem wrong? Disagree honestly. Don't be sycophantic.

---

## During Implementation

### Simplicity

- No features beyond what was requested
- No abstractions for single-use code
- No unrequested "flexibility" or "configurability"
- No error handling for impossible scenarios
- 200 lines that could be 50? Rewrite.

### Surgical Changes

- Don't "improve" adjacent code, comments, or formatting
- Don't refactor things that aren't broken
- Match the existing style, even if you'd do it differently
- Unrelated dead code noticed? Mention it — don't delete
- Remove ONLY imports/variables/functions that YOUR changes left orphaned
- Don't remove pre-existing dead code unless asked

### Test Integrity

- NEVER weaken an existing test assertion to make it pass
- NEVER delete a test to reduce the failure count
- NEVER use the framework's skip/disable/pending mechanism to bypass a failing test
- NEVER modify tests written in the RED phase during the GREEN phase
- If a test is genuinely wrong, STOP and confirm with the user before changing it
- Tests are the spec — the implementation conforms to the tests, not the other way around

### Goal-Oriented

- Turn vague tasks into verifiable goals
- Multi-step work? State a brief plan with verification checkpoints
- Every changed line should trace directly to the user's request

---

## After Each Change

Ask: "Would a senior engineer call this overcomplicated?"  
If yes → simplify before proceeding.

## Protheus Appendix — Mandatory Rules

These rules apply to **all** AdvPL/TLPP code generated, modified, or reviewed. No exceptions.

### Absolute prohibitions

| Forbidden | Reason | Alternative |
|---|---|---|
| `Function` (public) | Reserved for the standard product — customizations can't use it | `User Function` (public with `U_` prefix) or `Static Function` (file-private) |
| `IIF()` / inline `IF()` | SonarQube CA4000 — hurts readability and testability | Explicit `If/Else/EndIf` block |
| `cFilial` as a variable | Reserved system variable | `cFilAux`, `cFilBkp`, `cFilSA1`, etc., or `xFilial('XXX')` |
| `ConOut()`, `OutErr()`, `?` | SonarQube CA1004 — no tracking | `FWLogMsg()` |
| `MSCREATE()`, `DBCREATE()`, `CRIATRAB()` | SonarQube CA1000 — forbidden in Cloud/SmartERP | `FWTemporaryTable` |
| `FwFreeObj()`, `FreeObj()`, `FwFreeArray()` | Incomplete resource cleanup | The class's own `Destroy()` method |
| UI inside a transaction | SonarQube CA1002 — blocks multi-user, can cause deadlock | Move UI to before/after `Begin/End Transaction` |
| SQL without `FWExecStatement` or `ChangeQuery()` | SQL Injection (CA2050) | `FWExecStatement` or `ChangeQuery()` |
| Legacy includes | Obsolete — see table below | `#include "totvs.ch"` |
| `GetMV()`, `SuperGetMV()`, `ExistBlock()`, `Type()`, `Pergunte()` inside loops | Performance — slow calls per iteration | Cache the value before the loop |

### Mandatory includes

- **AdvPL**: `#include "totvs.ch"`
- **TLPP**: `#include "tlpp-core.th"` as the first include, then `#include "totvs.ch"`
- **NEVER** use: `protheus.ch`, `Ap5Mail.ch`, `ApWizard.ch`, `FileIO.ch`, `Font.ch`, `ParmType.ch`, `RWMake.ch`

### Mandatory SQL filters

Every `SELECT` on a Protheus table MUST include:
1. `D_E_L_E_T_ = ' '` — Protheus soft-delete
2. `XX_FILIAL = xFilial('XXX')` — branch filter
3. `%nolock%` on read queries — avoids unnecessary locks
4. `RetSqlName('XXX')` to get the physical table name (e.g., `SA1010`)

### Encoding

- **Every** `.prw`, `.prg`, `.prx`, `.tlpp`, `.ch`, `.aph` source file MUST be in **CP-1252 (Windows-1252)**
- Files generated by AI are in UTF-8 — convert with the `utf8-to-cp1252-conversion` skill before compiling
- Accented characters in UTF-8 silently corrupt at compile time

### Namespace in TLPP

```tlpp
// CORRECT: declare using namespace, use the class directly
using namespace totvs.framework.structure.interface
::oInterface := BuildContract():new("MATA010")

// WRONG: full namespace in the call
::oInterface := totvs.framework.structure.interface.BuildContract():new("MATA010")
```

### TLPP class pattern

- Access modifiers (`public`, `private`, `protected`, `static`) ONLY on the **declaration** inside the `class/endclass` block
- Implementations OUTSIDE `endclass` NEVER carry a modifier — only `method NomeMetodo(...) class NomeClasse`
- Inheritance uses `from`, not `extends`, `inherits` or `of`
- The constructor of an inherited class ALWAYS calls `_Super:new()`

### SonarQube Compliance

All code must comply with SonarQube rules for AdvPL/TLPP:
- Do not introduce CRITICAL or MAJOR violations
- INFO security rules (CA2050, CA2051, CA2052) are also mandatory
- Before generating or reviewing code, read the SonarQube rules reference file in the repository
